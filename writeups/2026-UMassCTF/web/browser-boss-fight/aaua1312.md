# 문제명

## 문제 설명

> This familiar brick castle is hiding something... can you break in and defeat the Koopa King?

- http://browser-boss-fight.web.ctf.umasscybersec.org:48003

## 풀이

### 분석
웹 페이지에 접속하면 key 입력창이 존재하며, 입력값을 검증하는 과정에서 클라이언트 측 스크립트가 동작하고 있다. 개발자 도구의 Network 탭을 통해 요청을 분석한 결과, key 입력은 /password-attempt로 POST 요청이 전송되며, 응답 헤더에 힌트가 포함되어 있음을 확인했다.


### 취약점

Client-side Validation 우회: key 입력 검증이 브라우저에서 수행되므로, 직접 요청을 전송하여 검증을 우회 가능
Cookie Manipulation: 인증 상태 및 게임 진행 상태를 쿠키(hasAxe)로 관리하며, 이를 클라이언트에서 수정 가능
Information Disclosure (HTTP Header): 서버 응답 헤더에 key 값에 대한 힌트 노출

### 익스플로잇

1. key 입력 요청 분석

    Network 탭에서 /password-attempt 요청 확인
    → 응답 헤더에 힌트 확인: King Koopa, if you forget the key, check under_the_doormat!
2. Client-side 검증 우회(gpt 도움)
    브라우저 콘솔을 이용하여 직접 POST 요청 전송
    fetch("/password-attempt", {
    method: "POST",
    headers: {
        "Content-Type": "application/x-www-form-urlencoded"
    },
    body: "key=under_the_doormat"
    });
    → /bowsers_castle.html 접근
3. 쿠키 확인
    I don't know how you got in, but you can't possibly defeat me! I removed the axe! 라는 쿠파의 말을 보고 보이지 않는 axe 클릭 시도 -> 실패
    axe 이름을 가진 쿠키 검색 시도 -> hasAxe=false 
4. 쿠키 변조 후 새로고침
    hasAxe=true 변경 후 새로고침 -> 플래그 등장
