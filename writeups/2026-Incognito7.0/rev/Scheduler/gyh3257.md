## 풀이

### 취약점

Ghidra로 디컴파일 후 핵심 로직을 파악했다.

**페이로드 함수 (4종)**

각 타입은 공유 배열(`DAT_00717060`, int 64개)에 값을 16개씩 쓴다.

| 타입 | 쓰는 값 | busy loop |
|------|---------|-----------|
| 1 (X77) | `0x13371337` | 50,000,000 iter |
| 2 (Y88) | `0xdeadbeef` | 75,000,000 iter |
| 3 (Z99) | `0xcafebabe` | 100,000,000 iter |
| 4 (W66) | `0xbaadf00d` | 125,000,000 iter |

**플래그 출력 조건**

```c
// 조건 1: 모든 burst time이 서로 다를 것 (unique & non-zero)
// 조건 2: hash(배열) == 0xe903d23a
if (unique_burst && hash_match) {
    system("cat flag.txt");  // "[+] 0xHASH_MATCH: ..."
}
```

**해시 함수 분석**

```c
uint hash(int *arr, int n) {
    uint sum = 0x811c9dc5;
    uint xor = 0;
    int transitions = 0;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
        xor ^= (arr[i] >> 2) * 0x1000193;
        if (i > 0 && arr[i] != arr[i-1]) transitions++;
    }
    uint result = sum ^ xor;
    if (transitions > 44) result ^= 0x1337beef;
    return result;
}
```

4가지 값이 각 16번씩 등장하므로 `sum`과 `xor`은 순서에 무관하다.
직접 계산하면:

```
sum = 0xFA346CD5,  xor = 0x00000000
→ result(transitions ≤ 44) = 0xFA346CD5  ≠ 0xe903d23a
→ result(transitions > 44) = 0xFA346CD5 ^ 0x1337BEEF = 0xE903D23A  ✓
```

**결론: 배열의 연속 전환 횟수가 44회를 초과해야 한다.**

- 블록 실행(스레드 순차): transitions = 3 → 실패
- 라운드로빈 인터리브: transitions = 63 → 성공

**신호 기반 선점형 스케줄러**

`FUN_004f7656`에서 SIGALRM 타이머(10ms 간격)와 핸들러를 등록한다.

```c
sigaction(SIGALRM, FUN_004f7455, NULL);
setitimer(/* interval= */ 10000 µs);
```

SIGALRM 발생 시 `FUN_004f7455`(스케줄러 핸들러)가 호출된다.
- 현재 스레드의 burst 감소 → 0 도달 시 DONE
- 우선순위 최솟값(= 가장 높은 우선순위) 스레드로 `swapcontext`
- **동일 우선순위** → `(현재+1) % n` 부터 탐색 = 라운드로빈

**깨진 Mutex (`FUN_004f7f27`)**

```c
void acquire() {
    do {} while (flag == 1);          // 스핀 대기
    for (i = 0; i < 50000000; i++) {} // 50M busy loop (비원자적!)
    flag = 1;
}
void release() { flag = 0; }
```

`do-while` 통과 후 `flag = 1` 세팅 사이에 50M 반복 busy loop이 있다.
이 구간(~50ms = 5 tick) 동안 SIGALRM이 발생해 다른 스레드로 전환되면,
다른 스레드도 `flag == 0`을 확인하고 spin을 빠져나온다.

→ **모든 스레드가 mutex를 동시에 획득**, 쓰기가 인터리브된다.

### 익스플로잇

**입력 조건 정리**

| 항목 | 값 |
|------|-----|
| 스레드 수 | 4 (타입 1·2·3·4 각 1개씩, 배열 64칸 채움) |
| 우선순위 | 모두 동일 (예: 50) → 라운드로빈 → transitions = 63 > 44 |
| burst time | unique, non-zero, 합계 ≤ 4000, 서버에서 충분히 커야 함 (≥ 200) |

```bash
printf "4\n1\n50\n200\n2\n50\n400\n3\n50\n600\n4\n50\n800\n" \
  | nc ctf.axiosiiitl.dev 1337
```

import subprocess, re

payload = "4\n1\n50\n200\n2\n50\n400\n3\n50\n600\n4\n50\n800\n"
r = subprocess.run(
    ["nc", "ctf.axiosiiitl.dev", "1337"],

print(re.search(r'IIITL\{[^}]+\}', r.stdout + r.stderr).group())

## 배운 점

- 신호 기반 선점형 스케줄러(SIGALRM + swapcontext)를 직접 구현한 바이너리 분석
- 비원자적 스핀락의 race condition이 의도적으로 삽입된 취약점으로 활용됨
- 해시 역산으로 필요한 배열 패턴(transitions > 44)을 수학적으로 도출
